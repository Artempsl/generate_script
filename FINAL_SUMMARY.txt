ВОПРОС 3: ReAct логика или строгая последовательность?
───────────────────────────────────────────────────────────────────────

⚠️ ОТВЕТ: Это СТРОГАЯ ПОСЛЕДОВАТЕЛЬНОСТЬ, а не полноценный ReAct!

ТЕКУЩАЯ РЕАЛИЗАЦИЯ:
-------------------
Тип: Linear Pipeline + Validation Loop (упрощенный ReAct)

Последовательность:
  retrieve → web_search → synthesize → generate_outline → generate_script
    ↓ always
  validate → (conditional) regenerate → validate
    ↓ if valid OR max_iterations
  END

Характеристики:
✓ Фиксированная последовательность (7 шагов всегда в одном порядке)
✓ ОДИН условный переход: should_regenerate()
✓ Нет динамической маршрутизации (агент НЕ выбирает, какой инструмент использовать)
✓ Нет reasoning traces ("почему я выбрал этот инструмент")
✓ Нет backtracking (нельзя вернуться к retrieve или synthesize)

Условие (ЕДИНСТВЕННОЕ):
  if script valid (90-110%) OR iterations >= 3:
      → END
  else:
      → REGENERATE → back to VALIDATE


ПОЛНОЦЕННЫЙ ReAct ВКЛЮЧАЛ БЫ:
-----------------------------
✗ Observation → [LLM decides next step] → Action → Observation (цикл)
✗ Динамический выбор инструментов на каждом шаге
✗ Reasoning: "I should use retrieve because..."
✗ Множественные ветвления (не только validate)
✗ Возможность пропускать/повторять произвольные шаги
✗ Backtracking к предыдущим состояниям

Пример полноценного ReAct:
  "I read the task → reasoning: need more context → action: retrieve
   → observation: context insufficient → reasoning: need web search 
   → action: web_search → observation: enough data → action: generate"


ПО ЧЕМУ ТАК СДЕЛАНО?
---------------------
✓ Предсказуемость: всегда одинаковое время выполнения
✓ Бюджет токенов: фиксированный, нет runaway costs
✓ Гарантия завершения: максимум 3 итерации, нет бесконечных циклов
✓ Легкость отладки: 7 шагов всегда в одном порядке
✓ Production-ready: стабилен для продакшена
✓ n8n-совместимость: фиксированные узлы workflow
✓ Детерминизм: одинаковые входные данные → одинаковый flow


КОГДА НУЖЕН ПОЛНЫЙ ReAct?
--------------------------
✗ Research tasks (исследование неизвестного)
✗ Complex problem-solving (многошаговые решения)
✗ Dynamic workflows (когда не знаешь заранее, какие шаги нужны)
✗ Long-running tasks (часы/дни работы)

Ваш случай:
✓ Генерация скриптов по шаблону → Linear Pipeline лучше!
